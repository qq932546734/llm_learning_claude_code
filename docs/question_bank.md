# 问题知识库（Question Bank）

> 作用：沉淀可复习的高价值问题，按知识点聚合，而不是按时间散落。
> v1 原则：仅收录 LLM 学习相关问题；`提问模式` 默认写入，`学习模式` 仅高价值问题写入。

## 使用规则
- 开场可进入 `学习模式` 或 `提问模式`。
- `学习模式`：提问继续按 `[记录候选]` 机制处理；会话结束时再判断是否进入知识库。
- `提问模式`：默认执行“回答 -> 总结 -> 归类 -> 回写 -> 是否加入复习”。
- 本文档负责“问题沉淀与复习入口”，不替代 `docs/question_analysis.md` 的认知画像功能。

## 记录模板

```yaml
问题ID: Qxxx
日期: YYYY-MM-DD
模块: Mxx
知识点: Xxx
问题: ...
总结: ...
状态: 新收录 / 会但不稳 / 已掌握
复习建议: 24小时内 / 3天内 / 7天内 / 暂不安排
来源模式: 学习模式 / 提问模式
```

## 复习使用方式
- 每次学习开场，优先从最近 24 小时且 `复习建议` 不为 `暂不安排` 的新收录问题中抽 1-2 个做热身。
- 同一知识点下出现多个问题时，尽量聚合到同一小节，避免重复散落。
- 若某问题连续两次复习都能稳定口述，可将状态从 `新收录` 调整为 `已掌握`。

```yaml
总结: 当前主流 decoder-only LLM 更常见的是 pre-norm 风格，而且很多模型实际用的是 RMSNorm 而不是经典 LayerNorm；主因是大模型深层训练时 pre-norm 更稳、梯度路径更顺。与此同时，Gemma 2 等新一些模型也出现了 pre + post 的混合做法，说明 post-norm 不是完全消失，而是通常不再作为默认基础形态。
```

## M08 人类反馈强化学习 (RLHF)

### Q078

```yaml
问题ID: Q078
日期: 2026-03-23
模块: M08
知识点: RLHF-PPO 中 KL penalty 的放置位置
问题: "RLHF 里的 KL penalty 到底加在哪？"
总结: `RLHF-PPO` 里，`KL penalty` 通常不是直接塞进 `clip loss` 公式，而是先按 token 计算当前 `policy` 相对 `reference` 的 log-prob 差，再乘 `-beta` 并到 reward 侧做 shaping；随后用这条 shaped reward 序列去算 `return / advantage`，最后再进入 `PPO clip loss`。因此它本质上是“先加在 reward 侧，再通过 advantage 间接影响更新”。
状态: 新收录
复习建议: 24小时内
来源模式: 学习模式
```

### Q079

```yaml
问题ID: Q079
日期: 2026-03-23
模块: M08
知识点: reference model / reference policy 的层级区别
问题: "Reference model 和 Reference policy 有什么区别吗？"
总结: `reference policy` 是策略/分布层面的概念，表示给定上下文后对下一个 token 的概率分布；`reference model` 是实现这套分布的具体冻结模型。在 `DPO / RLHF` 讨论里两者往往一一对应，但前者强调抽象的策略分布，后者强调工程实现中的那份具体模型。
```

### Q080

```yaml
问题ID: Q080
日期: 2026-03-24
模块: M08
知识点: reward shaping / shaped reward 的含义
问题: "做 shaping 是啥意思？shaped reward 是啥？"
总结: `reward shaping` 可以理解为把原始任务奖励加工成更适合训练的奖励信号。在 `RLHF-PPO` 里，原始奖励通常是序列末尾由 `reward model` 给出的整体分数，而 `shaped reward` 是训练时真正拿来算 `return / advantage` 的奖励序列：每个 token 上都会叠加相对 `reference policy` 的 `KL penalty`，末尾位置再叠加任务奖励。因此可记成“`shaped reward = 原始任务奖励 + 训练时额外加入的约束/引导项`”。
状态: 新收录
复习建议: 24小时内
来源模式: 学习模式
```

```yaml
总结: tokenizer 不只影响表示效果，还会同时改变 `vocab size` 和平均序列长度。平均序列长度会影响 `prefill` 的 attention 计算与 prompt KV 写入，也会影响 `decode` 阶段的 KV cache 读取；而 `vocab size` 会影响输出层规模与带宽，因此最终会一起作用到 `TTFT / TPOT / token/s`，也就影响系统吞吐。
```

```yaml
知识点: continuous batching 的最小调度粒度
问题: "continuous batching的最小单位是输出一个Token呢，还是完成一个补全任务？"
总结: `continuous batching` 的调度粒度更接近 `decode step / iteration`，而不是整条 completion 任务。每一轮生成里，batch 中的活跃请求通常各推进一个 token；结束的请求退出、空槽位立刻补进新请求，所以它本质上是 step 级动态补位。`prefill` 不完全一样，可能按整段 prompt 或 `chunked prefill` 插入，但 decode 核心节奏仍是逐 step 补位。
```
