# E01: 评估与基准测试（Evaluation & Benchmarking）

## 为什么学
- 没有评估，模型“变好了”通常只是感觉，不是结论。
- 这一块直接决定你能否把“离线分数、线上体验、judge 打分、人类偏好”区分开。

## 前置依赖
- M09 高级对齐技术
- M13 多模态 LLM
- M14 Agent 与工具使用

## 诊断题
1. benchmark 分数高，为什么不等于产品一定更好用？
2. `LLM-as-a-judge` 最常见的偏差来自哪里？
3. 为什么需要同时做自动评估、judge 评估和人工评估？

## 核心知识点
- `核心必会`：评估要先区分“模型能力评估”“任务评估”“产品评估”，三者不是同一层。
- `核心必会`：benchmark 更像标准化测量工具，适合横向比较，但常受数据污染、任务代表性和分布偏移影响。
- `核心必会`：`LLM-as-a-judge` 常用于相对比较和批量筛选，但不等于客观真值，尤其容易受位置偏差、措辞偏差和偏好模板影响。
- `工程重点`：评估设计通常要覆盖离线集、线上日志、失败案例回放和回归测试，而不是只盯排行榜分数。
- `面试高频`：回答评估方案时，优先按“目标 -> 指标 -> 样本来源 -> judge/human -> 回归机制”展开。

## 易错点 / 边界条件
- 容易把单一 benchmark 当成总体能力代表。
- 容易把 judge 分数当成人类偏好的完全替代。
- 容易忽略训练/检索/提示修改后的评估集污染问题。

## 练习
- [ ] 为一个客服助手设计 1 套离线评估，至少包含正确性、格式稳定性和拒答边界。
- [ ] 比较“自动指标高了”和“用户体验更好了”之间还缺哪些验证环节。
- [ ] 场景题：如果 judge 评分上升但用户投诉变多，你会先排查什么？

## 面试高频问法
### 问法 1
问题：为什么 benchmark 分数高不等于产品一定更好用？
- 一句话答案：因为 benchmark 测的是特定数据分布下的任务表现，而产品体验还受场景覆盖、交互链路、延迟和安全边界影响。
- 展开框架：分布代表性 -> 离线 vs 在线 -> 指标覆盖不足 -> 产品约束。

### 问法 2
问题：`LLM-as-a-judge` 有什么用，又有什么坑？
- 一句话答案：它适合批量做相对比较和候选筛选，但会继承偏差，不能替代高价值场景下的人类校准。
- 展开框架：为什么有用 -> 常见偏差 -> 何时仍需 human eval -> 怎么做校准。

## 完成标准
- 诊断题正确率达到 80%。
- 能区分 benchmark、judge 和 human eval 的分工。
- 完成 1 道评估方案题。

- 与 M13 / M14 相连：多模态和 agent 的好坏更依赖任务化评估而不是单一分数。
