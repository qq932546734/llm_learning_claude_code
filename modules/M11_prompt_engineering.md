# M11: 提示工程进阶

## 为什么学
- 这是把模型能力转成稳定产品行为的最低成本手段。

## 前置依赖
- 基本 LLM 使用经验
- M07 / M14 的应用场景直觉

## 诊断题
1. 为什么示例（few-shot）有时比更长的说明更有效？
2. 结构化输出最常见的失败模式是什么？你会如何防御？

## 核心知识点
- `核心必会`：in-context learning 的行为模式与上下文预算权衡。
- `核心必会`：schema、模板、约束提示对输出稳定性的影响。
- `工程重点`：CoT / ReAct / 自我检查的适用场景与代价。

## 易错点 / 边界条件
- 容易把 prompt 当万能药，忽略模型本身能力上限。
- 容易给太长说明，反而稀释关键信号。

## 练习
- [ ] 为一个结构化抽取任务写一版 prompt，并给出失败兜底策略。

## 费曼输出
- 用 120 字解释：为什么 few-shot 示例常常比抽象规则更有力？

## 完成标准
- 诊断题正确率达到 80%。
- 完成 1 次费曼输出。
- 完成 1 个结构化输出 prompt 设计题。

## 后续关联
- 与 `M14` 相连：prompt 设计直接影响 agent 规划、工具调用和安全边界。
