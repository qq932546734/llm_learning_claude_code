# E05: 量化与服务栈（Quantization & Serving Stack）

## 为什么学
- 同一个模型能不能上线、成本差多少，很多时候首先取决于量化方案和 serving 框架。
- 这一块能把“模型性能”翻译成“吞吐、延迟、显存、工程复杂度”的选择题。

## 前置依赖
- M06 模型并行与效率
- M12 生产部署与推理优化

## 诊断题
1. 为什么量化不是“位宽越低越好”？
2. `vLLM / TGI / TensorRT-LLM` 的选型为什么不是只看速度？
3. 哪些任务或模型结构对量化更敏感？

## 核心知识点
- `核心必会`：量化的目标是用更低精度表示权重或激活，以换显存、带宽和吞吐收益，但会带来精度损失和工程限制。
- `核心必会`：常见路线至少要区分 `PTQ / QAT / weight-only / activation-aware / FP8` 这几类思路。
- `核心必会`：serving stack 的选型不只看 kernel 性能，还要看模型支持范围、动态批处理、生态兼容和部署复杂度。
- `工程重点`：更低位宽通常先改善显存和带宽，但未必自动改善端到端延迟；收益受 kernel、batching、模型结构和硬件支持影响。
- `工程重点`：框架选型时，`vLLM` 更常被拿来做通用在线 serving，`TGI` 偏 Hugging Face 生态整合，`TensorRT-LLM` 更偏极致性能和硬件绑定。

## 易错点 / 边界条件
- 容易把量化理解成“纯精度压缩”，忽略 kernel 和硬件支持。
- 容易把吞吐提升直接等同于单请求延迟也一定下降。
- 容易只记框架名字，不记适用边界和工程代价。

## 练习
- [ ] 比较一个 7B 聊天模型做 FP16、INT8、INT4 时，收益和风险分别更可能落在哪。
- [ ] 场景题：如果你要快速上线通用聊天服务，会优先看 `vLLM`、`TGI` 还是 `TensorRT-LLM`？说明判断依据。
- [ ] 设计题：如果量化后事实性明显下降，你会先回退位宽、做校准还是换量化路线？

## 面试高频问法
### 问法 1
问题：为什么量化不是位宽越低越好？
- 一句话答案：因为量化是在显存/带宽收益和精度/稳定性/实现复杂度之间做权衡，不同模型和硬件的最优点不同。
- 展开框架：收益项 -> 失真项 -> kernel 支持 -> 任务敏感性。

### 问法 2
问题：`vLLM / TGI / TensorRT-LLM` 怎么选？
- 一句话答案：先看模型与硬件约束，再看动态批处理和生态兼容，最后才是单点性能。
- 展开框架：使用场景 -> 支持范围 -> 调度能力 -> 运维成本 -> 性能上限。
