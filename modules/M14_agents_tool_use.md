# M14: Agent 与工具使用

## 为什么学
- 这是从“回答问题”走向“调用外部系统解决任务”的跃迁。
- 真正难的地方不在“会不会输出工具调用 JSON”，而在于系统边界、规划、纠偏和权限控制。

## 前置依赖
- M11 提示工程
- 基本 API / 工具调用概念

## 诊断题
1. 工具调用里“模型可控性”的主要风险来自哪里？
2. 什么时候不应该让模型自由选择工具？
3. 为什么 `schema follow` 不是强 agent 的充分条件？
4. observation 出错后，什么时候该 `re-plan` 而不是机械重试？

## 核心知识点
- `核心必会`：`function calling` 更像结构化调用意图，`tool use` 是真实工具执行，`planning` 决定是否调用、调哪个、顺序如何以及 observation 后是否改计划。
- `核心必会`：典型 agent 工作流是 `plan -> act -> observe -> re-plan`，不是“一次输出完整计划就结束”。
- `核心必会`：`schema follow` 只能说明格式正确，不等于决策正确、能纠偏、能处理系统边界。
- `工程重点`：高风险、高成本、强权限约束场景下，不应让模型自由选择工具，通常需要白名单、强规则或人工确认。
- `工程重点`：agent latency 不能只看 LLM latency，还要把多轮 `TTFT / TPOT`、tool latency 和往返轮数一起算进去。
- `工程重点`：提示注入、越权、错误执行链、误读 observation、调用成功但任务失败，都是常见系统风险。

## 易错点 / 边界条件
- 容易把 agent 看成“更聪明的 prompt”，忽略权限和状态管理。
- 容易把“会输出 schema”误当成“会做正确决策”。
- 容易把是否让模型自由选工具，简化成“模型知不知道答案”，而忽略风险、成本和权限。
- 容易把 `re-plan` 理解成“步骤多就重规划”，而真正触发点是 observation 让原计划前提失效。

## 面试高频问法
### 问法 1
问题：为什么 `schema follow` 不是强 agent 的充分条件？
- 一句话答案：因为强 agent 不只要格式对，还要做对决策、会根据 observation 纠偏，并知道什么时候不该行动。
- 展开框架：格式层 -> 决策层 -> observation / re-plan -> 系统边界。
- 常见追问点：那 `function calling` 提升的到底是什么？为什么很多 agent 失败不是败在 JSON 格式？

### 问法 2
问题：什么时候不该让模型自由选择工具？
- 一句话答案：当任务涉及高风险、高成本或强权限约束时，工具选择不该完全交给模型自由决策。
- 展开框架：风险等级 -> 成本约束 -> 权限边界 -> 白名单/人工确认。
- 常见追问点：工具选择为什么属于 planning？如果模型“知道答案”就可以不调工具吗？
