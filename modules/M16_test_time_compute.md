# M16: 测试时计算扩展（Test-Time Compute）

## 为什么学
- 这是理解“多想一会儿为什么可能更准、但也更贵”的关键模块。
- 面试里这一块最容易被问成“为什么不是免费提升”“什么时候值得用”“和换更大模型相比怎么选”。

## 前置依赖
- M10 解码策略
- 基本推理与采样直觉

## 诊断题
1. self-consistency 为什么可能提升正确率？代价是什么？
2. 什么时候“多想一会儿”会适得其反？
3. 什么任务更适合 test-time compute：开放聊天、数学推理、代码、检索问答？
4. 为什么 test-time compute 不能被理解成“稳定提升模型能力”的万能按钮？

## 核心知识点
- `核心必会`：`self-consistency`、反思、搜索、verifier/critic 辅助等方法，本质上都在用更多推理路径或更多计算预算换更高命中率。
- `核心必会`：test-time compute 改善的是“如何调用/搜索已有能力”，不等于稳定创造模型本来没有的能力。
- `核心必会`：收益最常出现在可验证、可比较、可筛选候选答案的任务中，比如数学、代码、部分规划任务。
- `工程重点`：主要成本是延迟、token 消耗、不稳定性和系统复杂度；没有 verifier 或明确选择准则时，额外思考很容易只是在放大噪声。
- `面试高频`：判断是否值得上 test-time compute，优先看“任务是否可验证、预算是否允许、收益是否稳定”。

## 易错点 / 边界条件
- 容易把“更多推理步数”误当成稳定收益。
- 容易忽略任务类型和验证机制对收益的决定作用。
- 容易把 self-consistency 理解成“多采样必然更准”，却忽略候选质量和聚合规则。
- 容易把“换更大模型”和“增加 test-time compute”当成同一种扩展手段。

## 练习
- [ ] 比较“增加 test-time compute”和“换更大模型”的取舍。
- [ ] 场景题：数学题助手、日常聊天助手、检索问答助手三者里，分别判断是否值得上 `self-consistency` 或 verifier。
- [ ] 设计题：如果 tree search 的收益不稳定，你会先排查任务不适配、评分器不可靠还是候选多样性不足？

### 学习产物
- 1 页方法卡：`self-consistency / reflection / search / verifier` 的适用条件与代价。
- 1 组错题：至少记录 3 个“更多步骤 != 更强能力”“可验证任务更适合”“没有筛选机制就可能白花钱”的边界点。
- 1 个练习产物：任务选型表或“更大模型 vs 更多测试时计算”的取舍说明。

## 费曼输出
- 用 120 字解释：为什么 test-time compute 不是免费的性能提升？

## 面试高频问法
### 问法 1
问题：为什么 test-time compute 不是免费的性能提升？
- 一句话答案：因为它是拿更多推理路径、更多 token 和更高延迟去换可能更高的正确率，而这个收益既不稳定，也强依赖任务是否可验证。
- 展开框架：额外计算在做什么 -> 成本项 -> 任务依赖 -> 为什么开放任务收益不稳。
- 常见追问点：什么时候它最值得？什么时候几乎没必要？

### 问法 2
问题：什么时候更倾向于加 test-time compute，而不是直接换更大模型？
- 一句话答案：当任务可验证、请求量不大、单次正确率价值高，而且已有模型接近可解时，更适合加 test-time compute；否则更大模型或更好训练数据往往更稳。
- 展开框架：任务验证性 -> 延迟预算 -> 请求规模 -> 系统复杂度 -> 收益稳定性。
- 常见追问点：`self-consistency` 和 verifier 各自适合什么？为什么日常聊天不常默认上复杂搜索？
