# E02: 安全与红队（Safety & Red Teaming）

## 为什么学
- 模型能力变强后，安全问题通常不是“会不会发生”，而是“什么时候暴露、代价多大”。
- 这一块决定你能否把越狱、提示注入、过度拒答和权限风险放到同一张图里。

## 前置依赖
- M09 高级对齐技术
- M14 Agent 与工具使用

## 诊断题
1. jailbreak 和 prompt injection 的本质区别是什么？
2. 为什么过度拒答也是一种安全失败？
3. 工具型 agent 为什么更需要防御纵深，而不只是更强模型？

## 核心知识点
- `核心必会`：安全问题至少要区分输入攻击、行为失控、工具越权和输出伤害四层。
- `核心必会`：`jailbreak` 更常针对模型行为边界，`prompt injection` 更常针对系统链路和上下文污染。
- `核心必会`：red teaming 的目标不是“证明系统绝对安全”，而是系统性暴露高风险失效模式。
- `工程重点`：高风险系统通常需要多层防线：输入过滤、权限隔离、工具白名单、执行前确认、日志审计和回滚。
- `工程重点`：安全与可用性有张力，过度拒答、误杀正常请求也属于系统质量问题。

## 易错点 / 边界条件
- 容易把安全理解成“多写一条系统 prompt”。
- 容易把攻击只理解成文本越狱，忽略工具调用和检索链路污染。
- 容易把“拒绝得多”误当成系统更安全。

## 面试高频问法
### 问法 1
问题：jailbreak 和 prompt injection 的区别是什么？
- 一句话答案：前者更像试图突破模型行为约束，后者更像污染系统上下文或外部输入，让模型误把恶意内容当可信指令。
- 展开框架：攻击面 -> 影响层 -> 防御位置 -> 为什么 agent 更脆弱。

### 问法 2
问题：为什么过度拒答也算安全问题？
- 一句话答案：因为安全目标不是一味减少输出，而是在风险可控的前提下维持有用性；过度拒答说明系统把 `harmless` 压得过头了。
- 展开框架：帮助性 vs 安全性 -> 误杀成本 -> 用户体验 -> 策略调优。
