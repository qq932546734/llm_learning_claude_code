# E04: RAG 与检索（RAG & Retrieval）

## 为什么学
- 很多真实系统并不是“全靠模型记住”，而是“先找，再读，再答”。
- 这一块是长上下文之外最常见的知识增强路线，也是系统设计题高频项。

## 前置依赖
- M05 长文本建模
- M12 生产部署与推理优化
- M14 Agent 与工具使用

## 诊断题
1. 为什么 RAG 不是“加个向量库”就结束？
2. 什么场景更适合 RAG，而不是原生长上下文？
3. 检索命中率高，为什么回答仍可能错？

## 核心知识点
- `核心必会`：RAG 至少包含切分、索引、召回、重排、上下文打包和生成六个环节。
- `核心必会`：RAG 的本质是把信息筛选放到模型外部；它和长上下文的主要区别，不在“能放多少文本”，而在“先筛还是少筛”。
- `核心必会`：回答错误不只可能来自模型，也可能来自召回失败、切分错误、排序错误或上下文拼装错误。
- `工程重点`：chunk 大小、overlap、embedding 模型、reranker 和 citation 设计都会直接影响效果与成本。
- `工程重点`：RAG 更适合知识更新频繁、证据可检索、输入文档规模大且无需全文整体推理的任务。

## 易错点 / 边界条件
- 容易把“检索到了相关文档”误当成“系统就能答对”。
- 容易忽略 reranking 和 context packing 对结果的影响。
- 容易把 RAG 与长上下文说成绝对替代关系。

- [ ] 场景题：合同审阅和公司制度问答，哪个更偏 RAG，哪个更偏长上下文？说明原因。

## 面试高频问法
### 问法 1
问题：RAG 和长上下文的本质区别是什么？
- 一句话答案：RAG 把信息筛选放在模型外，长上下文更多把筛选留在模型内。
- 展开框架：信息流位置 -> 成本差异 -> 任务适配 -> 失败模式。

### 问法 2
问题：为什么 RAG 不是加个向量库就结束？
- 一句话答案：因为真正决定效果的是整条检索链路，而不是单独的向量检索这一步。
- 展开框架：切分 -> 召回 -> rerank -> 上下文打包 -> 生成 -> citation/校验。
