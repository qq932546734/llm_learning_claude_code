# M13: 多模态 LLM

## 为什么学
- 多模态是主流模型能力扩展的必经路线。
- 这一块真正拉开差距的不是“知道有图像输入”，而是能否说清结构、成本、训练目标和评估边界。

## 前置依赖
- M03 表示学习
- M07 / M11 的任务接口直觉

## 诊断题
1. 为什么需要“视觉 tokenizer / patch embedding”？它和文本 tokenizer 类似吗？
2. 多模态对齐最难的点通常是什么（数据 / 目标 / 架构）？
3. 为什么多模态不能简单理解成“图片先转文字，再丢给普通 LLM”？
4. 为什么多模态评估不能只看最后答对？

## 核心知识点
- `核心必会`：常见结构是 `vision encoder -> projector / Q-Former -> LLM`；视觉 token 与文本 token 是接口兼容，不是本体相同。
- `核心必会`：`patch embedding / visual tokens` 的作用是把连续视觉输入变成可供语言模型消费的离散步长表示，但不等于把图片“翻译成文字”。
- `核心必会`：多模态 instruction tuning 教的是“基于视觉证据按指令回答”，不只是“会看图”。
- `工程重点`：`projector / Q-Former` 是跨模态桥接层，常是先训练或重点微调的接口部分。
- `工程重点`：多模态常先恶化 `TTFT`，因为前面多了一段视觉编码，同时还会引入更多视觉 token，使 `prefill` 更重。
- `面试高频`：评估多模态不能只看答对，还要看模型是否真的依赖视觉证据，可用遮图、关键区域扰动、反事实样本做验证。

## 易错点 / 边界条件
- 容易把图像 patch 当成文本 token 的完全类比，忽略视觉结构信息与离散文本完全不同。
- 容易把 `projector` 说成“把图片变成文本 token”，更准确的说法是“做桥接和表示对齐”。
- 容易把“图片问答答对了”直接当成系统设计成功，却忽略语言先验和数据偏置。
- 容易忽略多模态数据质量、证据标注和对齐噪声对效果的影响。

## 面试高频问法
### 问法 1
问题：为什么多模态不能简单做成“图片转文字，再交给普通 LLM”？
- 一句话答案：因为很多视觉结构和细节在先转成文字时就会丢失，而且多模态训练要学的不只是描述图像，而是把视觉证据和语言指令一起纳入推理。
- 展开框架：视觉信息损失 -> 接口桥接层的必要性 -> instruction tuning 的联合目标 -> 工程成本。
- 常见追问点：那 OCR 路线什么时候仍然有用？`projector` 到底在做什么？

### 问法 2
问题：为什么多模态评估不能只看最后答对？
- 一句话答案：因为模型可能靠语言先验、数据偏置或模板记忆答对，而不是真的使用了图像证据。
- 展开框架：答对的伪来源 -> 遮图/扰动/反事实验证 -> 证据依赖与任务完成的区别。
- 常见追问点：遮图后准确率下降很少说明什么？图片问答准确率高本身说明不了什么？
