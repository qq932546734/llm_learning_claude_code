# E06: 现代架构扩展（MoE / SSM / Mamba）

## 为什么学
- 这块解决的是“Transformer 之外还有哪些路线、它们各自在解决什么瓶颈”。
- 面试里常见追问不是细节推导，而是：为什么要它、它解决了什么、又牺牲了什么。

## 前置依赖
- M01 Transformer 架构深度解析
- M06 模型并行与效率
- M16 测试时计算扩展

## 诊断题
1. MoE 为什么能在不等比例增加每 token 计算的情况下扩模型容量？
2. SSM / Mamba 这类路线试图解决 Transformer 的哪类瓶颈？
3. 为什么“不是 Transformer”不等于“会全面替代 Transformer”？

## 核心知识点
- `核心必会`：MoE 的核心思想是条件计算，让不同 token 只激活部分专家，以更高容量换接近受控的每 token 计算。
- `核心必会`：MoE 的关键工程问题不只是 router 本身，还有负载均衡、通信开销和专家热点。
- `核心必会`：SSM / Mamba 这类路线试图用状态递推或线性复杂度机制改善长序列效率与扩展性。
- `工程重点`：现代架构路线大多是在“容量、并行效率、长序列成本、硬件友好性”之间换取不同优势，而不是无条件碾压 Transformer。
- `面试高频`：回答这类主题时，优先说“它在解决什么瓶颈 -> 代价是什么 -> 什么时候更适合”。

## 易错点 / 边界条件
- 容易把 MoE 只理解成“更大模型”，忽略 router 和通信成本。
- 容易把线性复杂度路线误解成“所有任务都会更快更强”。
- 容易把“有新架构”误解成“Transformer 快被淘汰”。

## 练习
- [ ] 比较 Dense Transformer 和 MoE 在“容量 / 每 token 计算 / 通信复杂度”上的取舍。
- [ ] 场景题：如果你的主要痛点是超长序列吞吐，会先关注 MoE 还是 SSM/Mamba？为什么？
- [ ] 用 4 句解释为什么新架构通常不是“全面替代”，而是“特定瓶颈下更有吸引力”。

## 面试高频问法
### 问法 1
问题：MoE 为什么看起来参数更多，但每 token 计算不一定等比例增加？
- 一句话答案：因为 MoE 通过 router 让每个 token 只激活少数专家，增加的是总容量，不一定增加同等比例的活跃计算。
- 展开框架：条件计算 -> 活跃专家数 -> 容量与计算分离 -> 负载均衡/通信代价。

### 问法 2
问题：SSM / Mamba 这类路线想解决 Transformer 的什么问题？
- 一句话答案：它们主要想缓解长序列建模和扩展时的复杂度、缓存或硬件效率问题，但也会引入新的建模和工程权衡。
- 展开框架：Transformer 痛点 -> 递推/状态空间直觉 -> 潜在收益 -> 为什么不必然全面替代。
